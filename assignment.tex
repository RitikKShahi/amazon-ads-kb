\section{The Big Picture}
% ══════════════════════════════════════════════════════════════════════

\begin{infobox}[\faIcon{lightbulb} What does this system do?]
Imagine you want to build a Wikipedia-like knowledge base about Amazon Ads, but you don't want to write it by hand. This system \textbf{automatically} reads web pages, extracts useful facts, and organizes them into clean, structured markdown files --- all without creating duplicate information, even if you run it a hundred times.
\end{infobox}

\bigskip

\bigskip

The pipeline has three stages. Think of it like an assembly line:

\begin{enumerate}[leftmargin=*, label=\textcolor{primary}{\textbf{Step \arabic*.}}]
    \item \textbf{The Hash Scout} fetches the web page and creates a digital fingerprint (SHA-256 hash). If the fingerprint matches what we saw last time, the pipeline \textit{immediately stops}. No wasted API calls. No duplicate data.
    \item \textbf{The Extractor Agent} (an LLM) reads the raw text and distills it into clean, atomic factual claims formatted as JSON. It \textit{reasons} about what is a real fact vs.\ marketing fluff.
    \item \textbf{The Merger Agent} (another LLM) takes those claims, compares them \textit{semantically} against the existing knowledge base, and only writes genuinely new information. It \textit{reasons} about whether two sentences mean the same thing.
\end{enumerate}

\section{Claude Code, MCP Tools \& Scripts}
% ══════════════════════════════════════════════════════════════════════

\subsection{Why Claude Code?}
Claude Code acts as the \textbf{autonomous orchestration engine}. It doesn't just run code --- it hosts our multi-agent system, manages MCP tool connections, and provides the subagent framework that lets us split ``fetching'' from ``reasoning'' from ``writing.''

\subsection{Model Context Protocol (MCP) Integration}
The Extractor agent is given \textit{agency} to choose its own scraping strategy via MCP:

\begin{center}
\renewcommand{\arraystretch}{1.4}
\begin{tabular}{@{} l l l @{}}
\toprule
\textbf{MCP Tool} & \textbf{When It's Used} & \textbf{Why} \\
\midrule
\faIcon{search} Tavily Search & Static pages, blogs & Fast, token-efficient Markdown \\
\faIcon{theater-masks} Playwright & SPAs, JS-rendered pages & Real headless browser, full DOM \\
\bottomrule
\end{tabular}
\end{center}

\subsection{What Scripts Do (Deterministic) vs.\ What Agents Do (Reasoning)}

\begin{center}
\renewcommand{\arraystretch}{1.4}
\begin{tabular}{@{} >{\bfseries}l p{5cm} p{5cm} @{}}
\toprule
& \textbf{\faIcon{cog} Scripts (Deterministic)} & \textbf{\faIcon{brain} Agents (LLM Reasoning)} \\
\midrule
Purpose & State management, validation, hashing & Understanding text, deduplication \\
Examples & \texttt{hash\_source.js}, \texttt{validate\_okf\_write.js} & Extractor, Merger \\
Guarantee & Always produces the same output & Uses judgment to handle ambiguity \\
\bottomrule
\end{tabular}
\end{center}

\begin{infobox}[\faIcon{key} Key Design Principle]
\textbf{Scripts handle determinism. Agents handle intelligence.} We never let an LLM manage state or file hashes --- that's too risky. But we also never ask a script to decide if two sentences mean the same thing --- that requires reasoning. Each tool does what it's best at.
\end{infobox}

\section{Repository Structure}

\begin{lstlisting}
amazon-ads-kb/
|-- .claude/
|   |-- agents/                  # Subagent prompts (extractor, merger)
|   +-- settings.json            # Tool permissions & pre-write hooks
|-- scripts/
|   |-- hash_source.js           # Change detection via SHA-256 hashing
|   |-- validate_okf_write.js    # Taxonomy enforcement gate
|   +-- verify_integrity.js      # Post-run duplicate scanner
|-- state/
|   |-- seed_sources.json        # Starting URLs to crawl
|   +-- sources.json             # Hash registry (the system's memory)
|-- knowledge/                   # The final OKF knowledge base
|-- fixtures/                    # Cached raw page snapshots
+-- assignment.tex               # This document
\end{lstlisting}

\begin{itemize}[leftmargin=*, label=\textcolor{accent}{\faIcon{check-circle}}]
    \item \textbf{\texttt{hash\_source.js}} --- The brain of idempotency. Strips Amazon's tracking pixels, computes stable hashes, and decides if extraction should proceed.
    \item \textbf{\texttt{validate\_okf\_write.js}} --- A pre-write hook that Claude Code runs before \textit{any} file is saved to \texttt{knowledge/}. It blocks writes to unauthorized directories and rejects files missing OKF frontmatter.
    \item \textbf{\texttt{verify\_integrity.js}} --- A post-run scanner that reads every concept file and mathematically proves zero duplicate claims exist.
\end{itemize}

\section{The Problem We Solved: Amazon's Invisible Trap}
% ══════════════════════════════════════════════════════════════════════

\begin{dangerbox}[\faIcon{bug} The Dynamic Render Problem]
During testing, our pipeline entered an \textbf{infinite extraction loop}. Every time we fetched the same Amazon page, the hash was different --- even though the actual content hadn't changed at all. The system kept thinking the page was ``new'' and re-extracted it endlessly.
\end{dangerbox}

\textbf{Root Cause:} Amazon injects invisible tracking pixels into every page response. These pixels contain a rotating session ID that changes on every single HTTP request:

\begin{lstlisting}
![](//fls-na.amazon.com/1/batch/1/OP/A381WA5GOSXMH7:133-9572056-7926158:1P0...)
\end{lstlisting}

Because this random string is embedded in the Markdown, the SHA-256 hash of the page was never identical twice.

\begin{successbox}[\faIcon{wrench} Our Fix]
We implemented a regex sanitizer inside \texttt{hash\_source.js} that strips all \texttt{fls-na} tracking pixels \textit{before} computing the hash. This immediately stabilized the pipeline and allowed our idempotency proofs to succeed.
\end{successbox}

\textbf{The Honest Gap:} While our regex works perfectly for Amazon, a production system would need a more robust library like Mozilla Readability for universal DOM sanitization.

\section{Proof of Safe Re-Runs (Idempotency)}
% ══════════════════════════════════════════════════════════════════════

The pipeline guarantees safe re-runs via two distinct layers of protection.

\subsection{Layer 1: Hash-Skipping}

\begin{infobox}[\faIcon{shield-alt} How It Works]
If a source's text has not changed since the last run, the pipeline \textbf{stops immediately} before invoking the expensive Extractor LLM. Zero tokens are spent. Zero files are modified.
\end{infobox}

\textbf{Proof Snapshot 1:} Running the full orchestrator against \texttt{about-api}:

Action taken: None. Skipped extraction. Zero file diffs to knowledge base.
\end{lstlisting}

\subsection{Layer 2: Semantic Merge-Safety}

\begin{infobox}[\faIcon{code-merge} How It Works]
Even if a source \textit{does} change, the Merger agent \textbf{semantically compares} incoming claims against existing knowledge. It only writes genuinely new information. Duplicate facts are silently discarded.
\end{infobox}
